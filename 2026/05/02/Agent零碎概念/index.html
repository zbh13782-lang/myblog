Function Calling,Tool call,MCP

前情提要

二者解决的都是模型要调用工具的问题，MCP主要解决的是调用不同语言，不同AI，不同框架的问题，也就是模型上下文协议，规定了调用格式的问题。

另外说一下传输协议的问题吧。

传输协议

本地MCP tool

本地就是用stdio，这是一种进程间管道通信协议（操作系统讲的）。

  • 客户端（LangChain MCP Adapter）把 MCP Server 当成子进程启动
  • 客户端 → 子进程：写 stdin
  • 子进程 → 客户端：写 stdout
  • 消息格式：每行一个 JSON‑RPC 2.0 消息

外部MCP tool

比如GitHub MCP Server，也是我项目(about:blank)里用到的那个，遵守的是 Streamable HTTP 协议，以前是http + sse，这是属于计算机网络知识。我会在网络部分的八股和Agent八股里面详细说的。

旧版：HTTP + SSE（Server‑Sent Events）

  • 客户端发 HTTP POST → 服务端
  • 服务端用 SSE 长连接推数据 → 客户端（单向流）

通常是两个端点：

/sse 用 GET 打开 SSE 长连接，服务端通过它推消息

/messages 用 POST 发送客户端到服务端的 JSON-RPC 消息

举个例子：

  1. 客户端 GET /sse
  2. 服务端保持这个 SSE 连接不关闭
  3. 服务端通过 SSE 发一个 endpoint 事件，告诉客户端 POST 地址
  4. 客户端 POST /messages 发送 initialize、tools/call 等 JSON-RPC 消息
  5. 服务端如果要回消息、推通知、发进度，就通过之前那条 SSE 长连接推给客户端

新版：Streamable HTTP

  • 一个统一 HTTP 端点，同时支持：
    • POST：客户端 → 服务端（请求）
    • GET+SSE：服务端 → 客户端（响应 / 流式推送）
  • 双向流、支持会话、多客户端、可重连 Model Context Protocol

通常有一个端点：/mcp

简单请求的例子：

  1. 客户端 POST /mcp
    body: {“jsonrpc”:“2.0”,“id”:1,“method”:“tools/list”}
  2. 服务端直接返回：
    Content-Type: application/json{“jsonrpc”:“2.0”,“id”:1,“result”:{…}}

流式请求的例子：

  1. 客户端 POST /mcp
    body: {“jsonrpc”:“2.0”,“id”:2,“method”:“tools/call”,…}
  2. 服务端返回：
    Content-Type: text/event-stream
    event: message
    data: {“jsonrpc”:“2.0”,“method”:“notifications/progress”,…}
    event: message
    data: {“jsonrpc”:“2.0”,“id”:2,“result”:{…}}

这时 SSE 只是这次 POST 响应的流式返回形式，而不是必须提前建立的一条专用 /sse 通道。

二者的比较：原来是专门要先让客户端发送一个GET请求，之后服务端建立一个SSE管道来进行后续的服务端到客户端通信，这中间如果断连了，服务端无法及时感知到。原来的http+sse更适合长连接情况。

无论是本地还是外部，都遵循了JSON-RPC2.0格式，这也是MCP存在的一个必要吧，同意了一切的工具格式。

SKILL

书接上回，讲到了MCP，那就紧接着说说skill吧，首先二者不算是同一层的东西，更像是上下层，skill教会模型具体的任务说明书，这个任务需要用到工具的时候，MCP给了模型工具的使用说明书，skill更像是一种全局策略。

渐进式加载（动态加载）

这是skill设计的巧妙之处，这就解决了太长的skill塞入prompt的问题，只需要把skill的name和简短的description加载到system prompt，就能实现模型的skill。

第一层：只看 Skill 元数据

关键的在于description，决定要不要用。

第二层：触发后读取 SKILL.md主体

正文应该像“控制面板”，不是知识库。它放：

  • 核心工作流
  • 必须遵守的规则
  • 资源导航
  • 什么时候读取哪个 reference
  • 什么时候运行哪个 script

第三层：按需读取 references、assets、scripts

比如：

description 负责触发，SKILL.md 负责调度，references/scripts/assets 负责细节。

这里不得不感慨Anthropic的牛比之处了，虽然MCP和SKILL现在看起来是历史的必然趋势，但是人家提出来并发扬了确实吊。

通信方式

方案 A：Agent 和沙箱共享挂载目录

文件通过挂载目录传递。

控制命令通过 API / JSON-RPC 传递。

适合本地 Agent,如果SKILL只是被读取，建议把SKILL挂载为只读。

方案 B：不共享文件系统，只走 API 传文件

适合多租户，但大文件传输麻烦。

方案 C：Skill 在 Host，只把需要的文件复制进沙箱

方案 D：Skill 完全在沙箱镜像中

COPY skills /skills

解决方案

Skill 不要长期只放在沙箱里；应该放在 Host 的 Skill Library 里，由 Agent 可控索引。真正执行时，再把当前需要的 Skill 只读挂载到沙箱。

整体架构

Host 保存完整 Skill Library。

Agent 在 Host 上做 Skill 发现和渐进式读取。

Sandbox 只负责执行当前任务。

每次只挂载一个 Skill 到 /skill。

执行完销毁 Sandbox。

Single-Agent和Multi-Agent

Single Agent

单Agent工作，优点是适合小工作，因为记忆有限的问题，在处理小问题的时候非常好用。

缺点就是一个Agent要是干大工作要累死（上下文爆炸），还是因为上下文的问题。

Multi Agent

A2A

Google推出的一个协议，解决多Agent之间的通信问题。

提到通信问题，不得不提一个经典面试题，也是被问烂了：

Agent怎么保证输出格式？

例如JSON。

1. response format

这是目前最稳健的方法。主流的大模型 API（如 OpenAI, DeepSeek, Qwen）都提供了强制 JSON 输出的参数。开启后，模型在底层生成 Token 时会被限制只能生成符合 JSON 语法的字符。

2. Prompt

你是一个数据提取器，请返回JSON格式。

3. Pydantic，TypedDict

分工问题

有了多Agent，就很好解决了单Agent的上下文问题了，比如某个任务需要很干净的上下文去做（搜索网页之类的），只需要子Agent把返回的结果和总结给到Leader，而不要关注你搜索到多少，搜索到多好。

同时，加入我自己的一些理解，一定不要让Agent自主决定去开一个子Agent干，而是要加入人为干预，写入system prompt，什么任务可以开子Agent，什么任务可以交给别的Agent来干。

举个例子：

  1. 父Agent传入了错误信息，那子Agent又不知道是错的，干脆一错到底。
  2. 比如code过程

Coder 写代码，说按计划写的

Tester 跑测试，说环境不完整

这也出现了one shot，zero shot，few shot，就比如前面讲的让agent输出json格式，可以用few shot来引导。

Context

其实就是模型的上下文记忆，这里比较重要的就是三层记忆架构 和 三层渐进式压缩记忆。

三层记忆架构

  • 短期记忆（当前对话）
  • 中期记忆（智能压缩）
  • 长期记忆（CLAUDE.md）

三层压缩架构

  • Layer 1: MicroCompact（微压缩） — 无 LLM 调用，纯规则驱动，极致轻量。

这一层并不依靠ai去总结内容，而是针对工具上下文。

系统定义了一个可压缩工具白名单（COMPACTABLE_TOOLS），仅针对如 Bash、Read、Grep、Glob 等产生大量标准输出的工具进行压缩处理；而对于 Edit、Write 等涉及核心状态变更的操作，其输出则被完整保留，以确保后续决策的准确性。这种“抓大放小”的策略，既控制了体积，又守住了安全底线。

  • Layer 2: Session Memory Compact（会话记忆压缩） — 基于已有会话记忆进行替换，零额外推理成本。

仅完整保留最近5轮左右的完整会话，这是一个有门槛的压缩触发。

  • Layer 3: Full LLM Compact（完全压缩） — 调用 LLM 生成结构化摘要，精度最高但成本也最高。

有了这三层之后，就可以完成自动压缩机制了。

Claude Code的策略是设定一个安全缓冲水位线（AUTOCOMPACT_BUFFER_TOKENS = 13,000）。当上下文窗口剩余空间低于这个阈值时，系统会自动介入判断是否需要压缩。

整个决策流程是一个典型的分级回退策略（Fallback Strategy）：

  • 首选快速路径： 首先尝试 Session Memory Compact。因为它不需要额外的 LLM 调用，速度最快、成本最低。如果满足触发条件（Token 数和消息数达标），立即执行。
  • 降级重型路径： 如果 SM Compact 不满足条件（例如记忆尚未生成）或压缩后仍无法满足要求，系统会自动回退到 Full LLM Compact，不惜成本地生成高质量摘要以保全任务继续运行。

Harness

Harness约束的不是AI本身的能力（大脑），AI依旧强大，约束的是AI的手脚（具体而言就是Tool的权限）。

有些工具需要约束危险命令（比如bash），有些需要用户的确认才可以执行，总的而言都可以约束在沙箱里。
